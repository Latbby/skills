# Agent Test

## 目标

轻量执行 `test-cases.md` 中已经定义的 HTTP 接口用例，产出可复核的实际证据。只负责测试和报告，不重新规划、不修复代码、不扩展为通用测试框架。

## 输入与安全前置

接收 `docs/agent-workflow/<需求标识>/` 目录路径，并读取 `requirements.md`、`spec.md`、`tasks.md`、`reviews.md` 和 `test-cases.md`。先校验五份文档的 `workflow_id` 全部存在、完全一致并与输入目录名一致；任一不一致都执行全局阻塞流程。

1. 确认 `requirements.md` 为 `confirmed`，`spec.md` 和 `test-cases.md` 为 `ready`，`tasks.md` 文档状态及全部任务均为 `completed`，并确认 `reviews.md` 元数据同时满足 `status: passed`、`final_code_review: passed`、`final_requirements_review: passed`，且 `implementation_head` 是实际提交；否则禁止发送请求，创建总体为 `blocked` 的 `test-results.md`，把全部已定义用例逐例标为 `blocked` 后停止。此时总数按 `test-cases.md` 的全部定义用例数计算，并写明未满足的字段和恢复条件。
2. 从用例、仓库配置、环境变量和用户当前请求中解析基础地址与认证来源。
3. 禁止把令牌、密码、Cookie、API Key 或个人数据原文写入命令输出和结果文档。
4. 除非当前请求或工作流文档明确授权，禁止向生产环境发送请求。
5. 不猜测缺失的地址、账号或凭据；缺少时把相关用例记为 `blocked`，继续执行不依赖它们的用例。
6. 不创建持久化测试脚本或安装依赖。优先使用仓库已有 HTTP 客户端，其次使用系统现有的 `curl`、`Invoke-RestMethod` 等工具。

## 执行顺序

按 `test-cases.md` 的依赖关系和编号顺序逐条执行：

1. 检查前置条件和依赖用例；
2. 准备符合真实业务约束的测试数据；
3. 发送一次请求并捕获状态码、必要响应头、响应体和可观察副作用；
4. 逐条比较预期状态码、响应断言和副作用断言；
5. 执行清理步骤并记录清理结果；
6. 立即把结果追加到 `test-results.md`。

只有网络超时、连接重置或服务临时不可用等明确瞬时错误可以重试一次。业务失败响应不得通过重试掩盖。

一个用例失败或阻塞后，继续执行所有不依赖它的用例；依赖失败前置状态的用例标记为 `blocked`，不要伪造请求。

## 结果格式

创建 `test-results.md`，顶部使用 YAML 元数据记录 `workflow_id`、`stage: test` 和总体 `status`，正文记录执行环境与脱敏基础地址。总体状态只允许 `passed`、`failed`、`blocked`。每个用例使用以下格式：

```markdown
## TC-001：<业务场景>

- 状态：passed | failed | blocked
- 测试目的：<验证的业务行为>
- 实际入参：<方法、路径、脱敏请求头、查询参数和请求体>
- 实际出参：<状态码、必要响应头、脱敏响应体和副作用>
- 预期结果：<来自用例的状态码及断言>
- 实际结果：<逐条断言结果>
- 清理结果：<成功、失败或不需要>
- 证据：<命令摘要、日志位置或响应片段；不得含秘密>
- 失败原因：<失败或阻塞时填写；通过时写“无”>
```

响应很大时只保存验证需要的字段和稳定证据，不复制无关内容。二进制响应记录类型、长度、摘要等可验证信息。

## 判定规则

- 所有断言满足且清理成功：`passed`。
- 请求已执行但任一断言不满足，或清理失败造成污染：`failed`。
- 因缺少环境、凭据、前置数据或上游依赖失败而未执行：`blocked`。
- 不得因实现“看起来正确”而判定通过，必须以实际 HTTP 结果为准。

## 完成汇总

执行完可运行用例后，汇总总数、通过、失败、阻塞和清理失败数量，并列出失败/阻塞编号。总数始终按 `test-cases.md` 中全部已定义用例计算。总体状态按以下优先级聚合：存在任一 `failed` 或清理失败则为 `failed`；否则存在任一 `blocked` 则为 `blocked`；全部用例通过才为 `passed`。向用户报告同一摘要及结果文件路径。

不得修改业务代码、测试用例定义或 Git 历史；发现缺陷只记录证据和对应需求。
