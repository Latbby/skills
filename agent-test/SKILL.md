# Agent Test

接收 `docs/agent-workflow/<需求标识>/`，只执行和报告 `test-cases.md` 中的 HTTP 用例。

## 前置与安全

读取五份工作流文档并校验 `workflow_id` 均存在、相同且与目录一致；需求须为 `confirmed`，规格和用例为 `ready`，任务文档及全部任务为 `completed`，评审须满足 `status: passed`、两个终审为 `passed` 且 `implementation_head` 为真实提交。

任一全局前置失败时禁止请求，创建总体 `blocked` 的 `test-results.md`，将全部定义用例标为 `blocked`，写明字段和恢复条件。

- 从用例、配置、环境变量和当前请求解析基础地址与认证，不猜测缺失值。
- 令牌、密码、Cookie、API Key 和个人数据必须脱敏；未明确授权时禁止请求生产环境。
- 缺少环境、账号或凭据时仅阻塞受影响用例，继续独立用例。
- 不创建持久脚本或安装依赖；依次使用仓库客户端、`curl`、`Invoke-RestMethod` 等现有工具。

## 执行

按依赖和编号逐例：检查前置；准备真实业务数据；请求一次并捕获状态码、必要响应头、响应体和副作用；比较全部断言；清理；立即追加结果。仅网络超时、连接重置、服务临时不可用可重试一次，业务失败不得重试掩盖。

失败或阻塞后继续无依赖用例；依赖失败状态的用例标为 `blocked`，不得伪造请求。

## 结果

`test-results.md` 顶部记录 `workflow_id`、`stage: test`、`status: passed | failed | blocked` 和脱敏环境。每例记录：

```markdown
## TC-001：<场景>
- 状态：passed | failed | blocked
- 测试目的：<业务行为>
- 实际入参：<方法、路径、脱敏头、参数和请求体>
- 实际出参：<状态码、必要头、脱敏响应和副作用>
- 预期结果：<用例断言>
- 实际结果：<逐条结果>
- 清理结果：<结果或“不需要”>
- 证据：<命令摘要、日志或必要片段>
- 失败原因：<原因或“无”>
```

大响应只保留验证字段和稳定证据；二进制记录类型、长度、摘要。全部断言且清理成功为 `passed`；已请求但断言不符或清理污染为 `failed`；因环境、凭据、数据或依赖未执行为 `blocked`。不得以代码观感代替 HTTP 证据。

总数始终等于全部定义用例数。总体优先级为：存在失败或清理失败则 `failed`；否则存在阻塞则 `blocked`；全部通过才 `passed`。最终报告总数、通过/失败/阻塞/清理失败数量、相关编号和结果路径。禁止修改业务代码、用例定义或 Git 历史。
