# Agent Impl

## 目标

把一个已规划工作流完整实现为经过验证和双轴评审的原子提交序列。一次只处理一个任务，不跳过评审，不把问题累积到最后。

## 输入与基线

接收 `docs/agent-workflow/<需求标识>/` 目录路径。

1. 读取 `requirements.md`、`spec.md`、`tasks.md`、`test-cases.md`，确认需求为 `confirmed`、规格和测试用例为 `ready`；首次执行时任务文档必须为 `ready`，恢复执行时可以为 `in_progress` 或 `blocked`。
2. 校验四份文档 YAML 元数据中的 `workflow_id` 完全一致，并与输入目录名一致；任一缺失或不一致都按阻塞处理，禁止混用陈旧产物。
3. 读取适用的 `AGENTS.md`、仓库规范、构建与测试配置。
4. 确认当前目录是 Git 仓库且存在可解析的 `HEAD`，首次执行时记录为 `base_commit`；恢复执行时从 `reviews.md` 读取并验证已有 `base_commit`。
5. 检查工作区并区分已有用户改动。保护无关改动，禁止重置、覆盖或提交它们；若与任务修改范围冲突且无法安全隔离，按阻塞处理。
6. 使用仓库已配置的 Git 用户信息，禁止修改用户身份，禁止 push。

缺少任一前置产物时停止并说明缺失项，不向用户重新确认已确定的需求。

首次运行时创建 `reviews.md`，已有文件则保留历史并更新元数据。文档顶部使用以下机器可判定字段：

```yaml
---
workflow_id: <需求标识>
stage: impl
status: in_progress
base_commit: <首次执行前的提交>
implementation_head: pending
final_code_review: pending
final_requirements_review: pending
---
```

`status` 只允许 `in_progress`、`passed`、`blocked`；两个终审字段只允许 `pending`、`passed`、`blocked`。首次执行时还要把 `tasks.md` 文档元数据改为 `stage: impl`、`status: in_progress`。

## 逐任务循环

先恢复现场，再严格按依赖顺序执行：

1. 若恰有一个 `in_progress` 任务，结合工作区、最近提交和 `reviews.md` 判断已完成步骤，从该任务继续，不重复已验证工作。
2. 若存在 `blocked` 任务，逐项检查记录的恢复条件；条件已满足时改回 `pending`，未满足时保持阻塞并报告。禁止跳过它执行依赖任务。
3. 若同时存在多个 `in_progress`，或状态与提交证据互相矛盾，把工作流标记为 `blocked` 并记录修复条件，不擅自猜测。
4. 若全部任务已为 `completed`，但工作流因终审环境阻塞，恢复条件满足后保持 `tasks.md` 文档状态为 `completed`，把 `reviews.md` 改回 `in_progress` 并直接恢复最终全面复审。
5. 没有可恢复任务时，选择下一个依赖已完成的 `pending` 任务。

恢复具体开发任务后，把 `tasks.md` 和 `reviews.md` 文档状态统一改回 `in_progress`，再执行以下步骤。

### 1. 标记与限定范围

把任务状态改为 `in_progress`。只修改该任务列出的必要组件；发现范围外问题时记录到 `reviews.md`，不要顺手修复。

### 2. 处理复杂任务

当任务标记为 `complex` 时，必须启动子代理协助：

- 把工作拆为边界明确、互不重叠的调查或实现子任务；
- 给子代理提供规格、任务、适用规范和明确输出；
- 避免多个代理同时修改同一文件；
- 主代理检查子代理结果、整合改动并对最终实现负责。

若运行环境不支持子代理，将任务标记为 `blocked` 并记录恢复条件，不得把已判定为复杂的任务静默降级为单代理执行。

### 3. 实现与验证

复用现有工具函数、类型和组件，遵循当前仓库代码风格。为新增或修改的方法补充说明作用、入参和出参的必要注释，并为关键逻辑添加简洁单行注释。

完成最小实现后，执行任务指定的测试、构建、类型检查或静态检查。测试数据必须符合真实业务约束。失败时先诊断和修复根因，不得绕过检查或降低断言。

### 4. 双轴初审

由主代理在两个明确分离的检查轮次中评审当前任务改动：

- **代码轴**：检查正确性、异常处理、安全性、维护性、仓库规范、无关改动和测试缺口。
- **需求轴**：逐条对照 `requirements.md`、`spec.md` 和当前任务，检查缺失、部分实现、错误行为、未要求行为和范围膨胀。

把发现按 `blocking`、`major`、`minor` 分级并追加到 `reviews.md`。修复所有 `blocking` 和 `major` 发现，再重新运行受影响验证和对应评审；`minor` 可以保留，但必须说明不修复理由。

每次验证记录：目的、入参、出参、预期结果和实际结果。

### 5. 完成与提交

只有实现、验证和双轴初审均通过后，才执行：

1. 将任务状态改为 `completed`；如果这是最后一个未完成任务，同时把 `tasks.md` 文档元数据改为 `status: completed`；
2. 在 `reviews.md` 写入任务结论和验证摘要；
3. 检查 diff，只暂存该任务代码、测试及工作流文档；
4. 检查暂存内容不含凭据或无关文件；
5. 创建一个中文原子提交，例如 `feat(订单): 完成 T001 创建接口` 或 `fix(权限): 完成 T003 越权校验`。

提交失败时保留安全的工作区状态并修复原因。不得使用破坏性 Git 命令。

## 最终全面复审

全部任务 `completed` 后，确认 `tasks.md` 文档元数据已经在最后一个任务提交中变为 `status: completed`。把当前提交写入 `reviews.md` 的 `implementation_head`。以 `base_commit...implementation_head` 为固定实现评审范围，同时启动两个相互独立的子代理。若运行环境无法同时提供两个独立子代理，把工作流标记为 `blocked` 并记录恢复条件：

- **代码评审子代理**：读取完整 diff 和仓库规范，只评审代码轴。
- **需求评审子代理**：读取完整 diff、需求、规格和任务，只评审需求轴。

要求两者给出带文件与证据的分级发现，避免互相共享中间结论。主代理把结果并列写入 `reviews.md`，不让一个轴的结论掩盖另一个轴。

若存在 `blocking` 或 `major` 发现，继续修复、运行相关验证、创建独立中文修复提交，把新提交更新为 `implementation_head`，并再次执行两个独立复审，直到没有阻断发现。最后运行一次与改动范围相称的完整验证，并记录五要素结果。

评审通过后，把最终双轴结论和验证结果追加到 `reviews.md`，并把元数据 `status`、`final_code_review`、`final_requirements_review` 全部改为 `passed`。创建一个中文文档提交，例如 `docs(评审): 记录最终双轴复审结果`。该提交只允许包含工作流记录，不改变实现快照，因此明确排除在 `base_commit...implementation_head` 的实现评审范围外，不因记录评审而触发新的复审循环。

## 允许停止的特殊阻塞

除以下情况外不得中途停下：

- 缺少无法自行取得的必要权限、凭据或外部服务；
- 下一步需要未经授权的破坏性、生产或对外操作；
- 已确认需求存在无法同时满足的硬冲突；
- 环境或依赖连续三次出现同一外部失败，且没有本地替代验证；
- 无关用户改动与任务产生无法安全隔离的冲突；
- 复杂任务必须使用子代理但运行环境不提供该能力。
- 最终全面复审无法取得两个独立子代理执行环境。

发生阻塞时，把当前任务标记为 `blocked`（尚未进入具体任务时不虚构任务状态），把 `tasks.md` 文档状态和 `reviews.md` 的 `status` 改为 `blocked`，在正文写明证据、已尝试动作和唯一恢复条件，然后报告；恢复条件满足后按“逐任务循环”的恢复规则继续。不要用普通不确定性作为停止理由。

## 完成条件

只有以下条件全部满足才宣布完成：所有任务均为 `completed`；每项均有对应提交和初审记录；最终两个评审轴无阻断发现且结论已提交；最终验证通过；工作区不包含本技能产生的未提交改动；从未 push。
