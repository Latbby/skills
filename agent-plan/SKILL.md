# Agent Plan

接收 `docs/agent-workflow/<需求标识>/`。确认 `requirements.md` 的 `workflow_id` 与目录一致、`stage: ask`、`status: confirmed`，再读取适用的 `AGENTS.md`、代码、配置、类型、测试和文档。前置缺失时说明后停止，不询问或猜测其他目录。

## 决策原则

规划阶段不提问。未规定的细节依次选择：复用现有设计；最小改动、最低风险、最易验证和回滚；保持公开接口与持久化格式兼容；不预建未来能力。将影响实现的推导写入规格的“假设与自动决策”。

确认需求存在硬冲突时，完整重写下述三份产物，统一写入 `status: blocked`、冲突证据和恢复条件，任务及用例留空，然后停止。

## 输出

完整重写以下文件，三者使用同一 `workflow_id`、`stage: plan` 和 `status: ready | blocked`。

### `spec.md`

包含：概要与成功标准、非目标、当前实现证据、方案与受影响模块、接口/类型/数据/配置契约、数据流、校验与错误、边界行为、兼容与迁移、安全/性能/可观测性（仅相关项）、假设与自动决策、可验证验收标准。

接口契约明确输入、输出、错误语义和兼容要求。只固化需求或仓库约定支持的行为；无事实依据且不影响验收的细节沿用框架默认并标为“非契约行为”，影响核心目标时按上述原则裁决并记录。

### `tasks.md`

按依赖顺序拆成可独立实现、评审、验证和提交的任务，相关测试随行为实现，不单独堆到最后：

```markdown
## T001：<单一交付目标>
- 状态：pending
- 依赖：无 | Txxx
- 复杂度：simple | complex
- 涉及组件：<必要路径>
- 实施说明：<可直接执行的行为>
- 需求依据：<需求/规格条目>
- 验证方式：<检查方式>
- 完成条件：<可观察结果>
```

跨三个以上独立组件、涉及安全/迁移/高风险兼容、可拆成独立子任务，或陌生代码需独立验证时标为 `complex`。

### `test-cases.md`

只为需求涉及的 HTTP 接口编写真实业务场景：

```markdown
## TC-001：<业务场景>
- 关联需求：<条目>
- 依赖用例：无 | TC-xxx
- 优先级：P0 | P1 | P2
- 前置条件：<环境、数据、账号>
- 认证：<来源或“不需要”，不写密钥>
- 请求：<方法、路径、参数、必要头和请求体>
- 测试数据：<真实约束数据>
- 预期状态码：<状态码>
- 响应断言：<稳定契约>
- 副作用断言：<变化或“无”>
- 清理：<方法或“无”>
```

按适用性覆盖成功、校验、认证、授权、不存在、冲突、幂等和分页；不为覆盖率制造无关用例，不断言偶然行为、未确认的精确文案、字段顺序或重复参数解析。

结束前验证需求映射完整、任务无循环依赖且均为 `pending`、用例可安全执行、无占位符或关键未决项。仅报告目录、文件列表、任务数和用例数。
