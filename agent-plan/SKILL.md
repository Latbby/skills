# Agent Plan

## 触发约束

仅在用户当前请求中明确点名 `agent-plan` 或手动执行对应命令时运行。不得因为需求已经确认、用户要求制定计划或仓库中存在工作流目录而自动触发；未被明确调用时，不读取或推进本技能工作流。

## 目标

把已确认需求转换成决策完整、可直接交给实现代理执行的开发包。整个阶段自动运行，不重新讨论 `agent-ask-me` 已确认的需求。

## 输入与前置检查

接收 `docs/agent-workflow/<需求标识>/` 目录路径，并读取其中的 `requirements.md`。

1. 确认元数据 `stage: ask`、`status: confirmed`。
2. 读取仓库根目录及相关子目录的 `AGENTS.md`、代码、配置、类型、测试和文档。
3. 若需求目录或确认状态缺失，记录缺少的前置条件并停止；不要询问用户，也不要猜测另一份需求目录。

## 自动决策规则

规划阶段禁止向用户提问。遇到未规定的实现细节时：

1. 优先复用现有模块、类型、工具函数、组件和测试模式；
2. 选择改动最小、风险最低、最易验证和回滚的方案；
3. 保持公开接口和持久化格式向后兼容；
4. 只设计当前需求需要的能力，不为假设中的未来需求扩展范围；
5. 把会影响实现的推导写入 `spec.md` 的“假设与自动决策”。

如果确认需求内部存在无法同时满足的硬冲突，必须完整重写 `spec.md`、`tasks.md`、`test-cases.md`：三者写入相同 `workflow_id` 和 `status: blocked`，正文只保留冲突证据与恢复条件，任务和用例列表为空。这样可以使既有 `ready` 产物立即失效。随后停止，不生成看似完整但不可执行的计划。

## 生成开发规格

创建或完整重写需求目录下的 `spec.md`，至少包含：

1. 元数据：`workflow_id`、`stage: plan`、`status: ready`；`status` 只允许 `ready`、`blocked`；
2. 概要与成功标准；
3. 非目标；
4. 当前实现与证据；
5. 总体方案和受影响模块；
6. 接口、类型、数据及配置契约；
7. 数据流与关键流程；
8. 校验、错误处理和边界行为；
9. 兼容性及迁移策略；
10. 安全、性能、可观测性要求，仅保留与需求相关的内容；
11. 假设与自动决策；
12. 可逐条验证的验收标准。

接口契约必须明确输入、输出、错误语义和兼容要求。不得编造仓库中不存在且需求不需要的接口。

只把已确认需求或仓库现有约定支持的行为固化为公开契约。若重复参数处理、精确错误文本、字段顺序等细节没有事实来源，也不影响验收，则明确沿用现有框架默认行为并标记为“非契约行为”，不要为其设计自定义规则或精确测试断言。若某个无来源细节会直接影响核心成功标准，则按自动决策规则选择最低风险方案，并在“假设与自动决策”中给出依据。

## 拆分开发任务

创建或完整重写 `tasks.md`。文档顶部写入固定元数据：

```yaml
---
workflow_id: <需求标识>
stage: plan
status: ready
---
```

文档级 `status` 在规划阶段只允许 `ready`、`blocked`。任务按依赖顺序排列，每项使用以下结构：

```markdown
## T001：<单一可交付目标>

- 状态：pending
- 依赖：无 | Txxx
- 复杂度：simple | complex
- 涉及组件：<模块或路径，控制在必要范围>
- 实施说明：<足以直接实现的行为说明>
- 需求依据：<requirements/spec 中的条目>
- 验证方式：<测试、构建、静态检查或手工验证>
- 完成条件：<可观察结果>
```

满足任一条件时标记 `complex`：跨越三个以上相对独立的组件；包含安全、迁移或高风险兼容逻辑；可拆成互不重叠的调查或实现子任务；现有代码陌生且需要独立验证。

每个任务必须能独立实现、评审、验证和提交。不要把“编写全部测试”单独放到最后；相关自动化测试应与对应行为在同一任务完成。

## 生成 HTTP 接口测试用例

创建或完整重写 `test-cases.md`。文档顶部写入与 `tasks.md` 相同的 `workflow_id`、`stage: plan`、`status: ready` 元数据；文档级 `status` 只允许 `ready`、`blocked`。只为需求实际涉及的 HTTP 接口生成用例，不扩展到 UI、CLI 或通用单元测试。

每个用例包含：

```markdown
## TC-001：<业务场景>

- 关联需求：<条目>
- 依赖用例：无 | TC-xxx
- 优先级：P0 | P1 | P2
- 前置条件：<服务、数据、账号和环境>
- 认证：<凭据来源或“不需要”；禁止写真实密钥>
- 请求：<方法、路径、查询参数、必要请求头和请求体>
- 测试数据：<符合真实业务约束的数据>
- 预期状态码：<HTTP 状态码>
- 响应断言：<字段、类型、值或错误结构>
- 副作用断言：<状态变化；无则写“无”>
- 清理：<恢复测试数据的方法；不需要则写“无”>
```

根据接口能力覆盖适用的成功、参数校验、未认证、无权限、资源不存在、冲突、幂等和分页场景。禁止为了凑覆盖率生成不适用用例。

测试断言只锁定需求或现有契约要求的状态码、字段和错误语义；不要把框架偶然行为、未确认的精确错误文本、响应字段顺序或重复参数解析方式写成验收条件。

## 完成检查

结束前验证：

- 三份输出在 YAML 元数据中使用同一 `workflow_id` 和可消费状态；
- 每条需求至少映射到规格、任务和适用测试；
- 任务没有循环依赖且状态均为 `pending`；
- 用例可以在不暴露凭据、不默认访问生产环境的前提下执行；
- 文档不存在占位符、未决问题或要求实现者自行选择的关键方案。

只返回需求目录、文件列表、任务数量和用例数量，不开始编码。
