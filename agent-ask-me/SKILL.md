# Agent Ask Me

将用户想法收敛为 `docs/agent-workflow/<需求标识>/requirements.md`，不生成规格、任务、代码或测试。标识优先采用用户输入，否则生成简短 kebab-case；目录冲突时追加序号。

## 文档契约

首次调查后创建文档，每次决策后立即更新：

```yaml
---
workflow_id: <需求标识>
title: <需求标题>
stage: ask
status: draft # draft | confirmed
---
```

正文持续维护：背景与问题、目标与成功标准、用户与场景、范围内、范围外、当前状态与证据、功能需求、非功能约束、验收标准、决策记录、假设与风险、未决事项。

## 执行流程

1. 读取相关 `AGENTS.md`、代码、配置、类型、接口、测试和文档。自行查证事实并记录来源，不向用户询问可调查的信息。
2. 识别需要裁决的事项，为每项准备 2–3 个互斥可行选项，依次比较目标匹配、仓库一致性、改动与验证成本、风险、兼容性和可回滚性。
3. 有唯一最优项时自动选择改动最小、风险最低、最易验证和回滚的方案。
4. 以下事项必须交给用户：最优项并列；数据丢失或不可逆操作；认证、授权、隐私、安全、合规；破坏公开接口、持久化格式或外部承诺；生产、显著成本、外部承诺；昂贵且难回滚的架构；实质改变目标、成功标准或范围。
5. 每轮只问一个问题。提问前先记录问题、选项、推荐项及理由，并把用户选择和最终选择写为“待回答”；收到回答后立即补全。
6. 目标、范围、约束、关键依赖、主要与边界行为、失败行为、验收标准及重大决策均明确，且未决事项为空时，将状态改为 `confirmed`，无需再次总体确认。

每项决策使用稳定递增编号，变更时新增记录并引用旧编号：

```markdown
### D-001：<标题>
- AI 问题：<待判断的问题>
- 上下文：<需求或仓库事实>
- 选项：A. <选项与影响>；B. <选项与影响>
- 推荐选项：A
- 推荐理由：<依据>
- 用户选择：<回答；自动裁决写“未询问”>
- 最终选择：<结果>
- 决策者：AI | 用户
- 决策理由：<依据；用户未说明则如实记录>
```

只有关键事实无法取得，或用户拒绝重大决策时保持 `draft`，在未决事项中记录原因和唯一恢复条件。完成后仅返回目录和需求摘要，不进入规划或实现。
