# Agent Code Review

对用户指定范围做独立评审。没有明确范围时，优先使用当前未提交 diff；若提供 `docs/agent-workflow/<需求标识>/`，按其 `base_commit...implementation_head` 评审并读取需求、规格和任务。范围仍无法确定时才询问用户。

## 前置

读取适用的 `AGENTS.md`、仓库规范和评审范围。工作流模式下校验 `workflow_id` 与目录一致、需求已确认、规格可用、实现头是实际提交；前置不成立则报告阻塞证据，不猜测范围。

保持只读：不修改代码、工作流状态、Git 历史或外部系统，不运行会改变业务数据的测试。可以执行只读检查、静态分析和安全的本地测试；测试后记录目的、入参、出参、预期和实际结果。

## 双轴评审

分两轮检查，避免需求结论掩盖代码缺陷：

1. **代码轴**：正确性、边界与异常、并发和数据一致性、安全与隐私、兼容性、性能、可维护性、仓库规范、测试缺口及范围外改动。
2. **需求轴**：逐条对照需求、规格和任务，检查遗漏、部分实现、错误语义、未要求行为和范围膨胀。没有需求文档时，以用户请求和可验证现有行为为依据，并明确证据限制。

只报告可复现、可定位、会影响正确性/安全/维护成本的发现，不把个人风格偏好当缺陷。每项按以下级别排序：

- `blocking`：会造成严重安全、数据损坏或无法交付；
- `major`：明确功能错误、需求缺失、兼容破坏或关键测试缺口；
- `minor`：局部风险或维护问题，不阻断交付。

## 输出

先列发现，按严重级别和影响排序，每项包含：

```markdown
### <级别>：<标题>
- 位置：<文件:行号或提交>
- 证据：<触发条件及观察>
- 影响：<用户或系统后果>
- 建议：<最小、低风险、可回滚的修复方向>
```

随后给出评审范围、代码轴结论、需求轴结论、已运行验证及残余风险。没有发现时明确写“未发现阻断问题”，但仍说明未覆盖区域；不得声称证明代码完全正确。

若工作流目录中已有 `reviews.md`，默认仅在回复中给出结果；只有用户明确要求写入时，才追加独立评审章节，且不改变其元数据状态。禁止修复、暂存、提交或 push。
